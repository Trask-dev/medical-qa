## 1. 架构概述

### 1.1 系统定位

医疗智能问答系统是一个基于大语言模型（LLM）的健康咨询平台，采用 **Master-Agent 多智能体协作架构**，通过结构化信息采集、循证知识检索和综合推理，为用户提供安全、准确、有依据的健康咨询服务。

### 1.2 核心架构特征

| 特征 | 描述 |
|------|------|
| **多Agent协作** | MasterAgent 统领问诊、搜索、诊断三大Agent协同工作 |
| **有状态工作流** | 基于 LangGraph 构建的状态图驱动对话流转 |
| **异步并行** | 问诊与搜索并行执行，不阻塞用户对话主线程 |
| **安全优先** | 多层安全护栏，急危重症实时识别并引导就医 |
| **模型可配置** | LLM基座支持热切换，无代码级模型变更 |
| **循证可溯源** | 所有医学建议附带引用来源，知识时效性自动管理 |

### 1.3 设计目标

```
┌──────────────────────────────────────────────────────┐
│                   系统设计目标                          │
├──────────────┬──────────────┬──────────────┬──────────┤

├──────────────┼──────────────┼──────────────┼──────────┤
│ 急危重症识别  │ 权威知识引用  │ 术语标准化    │ 模型热切换 │
│ 红旗症状拦截  │ 来源可追溯    │ JSON结构化   │ Agent扩展  │
│ 禁止AI确诊   │ 时效性管理    │ 病历完整性    │ 存储升级   │
└──────────────┴──────────────┴──────────────┴──────────┘
```

## 2. 核心设计理念

### 2.1 设计原则

#### 2.1.1 单一职责原则（SRP）

每个Agent只负责一个明确限定的职责领域：
- **MasterAgent** — 意图识别与路由分发，不做业务处理
- **问诊Agent** — 信息采集与结构化，不做诊断推理
- **搜索Agent** — 知识检索与过滤，不参与对话
- **诊断Agent** — 综合分析推理，不直接与用户交互

#### 2.1.2 关注点分离

```
        ┌─────────────┐
        │  用户交互层   │  ← 自然语言对话、流式输出
        ├─────────────┤
        │  业务流程层   │  ← Agent编排、状态流转
        ├─────────────┤
        │  知识服务层   │  ← RAG检索、向量搜索
        ├─────────────┤
        │  安全护栏层   │  ← 内容审核、PII脱敏、红旗检测
        ├─────────────┤
        │  基础设施层   │  ← LLM、数据库、消息队列
        └─────────────┘
```

#### 2.1.3 防御性设计

- **输入假设最坏情况**：所有用户输入可能包含危险意图
- **输出强制安全校验**：LLM输出经JSON Schema校验后才可返回
- **降级策略**：任何组件异常都有兜底路径（引导就医）

### 2.2 架构决策记录（ADR）

| 决策ID | 决策内容 | 理由 | 替代方案 |
|--------|---------|------|---------|
| ADR-001 | 采用LangGraph而非自研状态机 | 原生支持有状态循环工作流，社区活跃 | 自研FSM、Temporal |
| ADR-002 | Master-Agent集中路由模式 | 统一意图识别，便于安全拦截 | 点对点Agent通信 |
| ADR-003 | 黑板模式共享状态 | Agent间解耦，状态可追溯审计 | 消息队列直连 |
| ADR-004 | 问诊与搜索异步并行 | 问诊不阻塞搜索，用户感知延迟最小 | 串行瀑布式 |
| ADR-005 | 初期PostgreSQL + pgvector，预留分布式升级路径 | pip安装即用，零配置启动 | ChromaDB、Pgvector |

## 4. Agent 体系设计

### 4.1 Agent 总体架构

系统采用 **Master-Agent 协作模型**，由一个主控Agent和两个专业Agent组成。其中问诊Agent内部采用 Twin-Track 并行架构：

```
                         ┌──────────────────┐
                         │   MasterAgent    │
                         │   (总指挥/分诊台)  │
                         └────────┬─────────┘
                                  │
                    ┌─────────────┼─────────────┐
                    │             │             │
             ┌──────▼──────┐      │      ┌──────▼──────┐
             │  问诊Agent   │      │      │  诊断Agent   │
             │ (Twin-Track) │      │      │ (综合推理)   │
             │              │      │      └─────────────┘
             │ Track A:框架  │      │
             │ Track B:搜索  │      │
             │   (并行)      │      │
             └──────────────┘      │
                                   │
                      ┌────────────▼────────────┐
                      │       安全护栏层         │
                      │  (L0过滤 + PII脱敏 +      │
                      │   内容合规 + 审计日志)     │
                      └─────────────────────────┘
```

> **架构变更 (v2.0)**: 原三Agent并行模型（问诊/搜索/诊断）已演进为**两阶段流水线 + Twin-Track并行**模型。搜索不再作为独立Agent，而是作为问诊Agent内部的 Track B 与 Track A（框架驱动采集）每轮并行触发。

### 4.2 MasterAgent（编排主控）

#### 4.2.1 职责定义

MasterAgent 是整个系统的中枢神经系统，负责意图识别、流程编排和安全兜底，**不参与任何业务逻辑处理**。

#### 4.2.2 核心能力

```
│  ┌──────────┐   ┌──────────┐   ┌──────────┐        │
│  │ 意图识别  │   │ 路由分发  │   │ 安全兜底  │        │

│  │ diagnosis│──→│→问诊Agent│   │ 红旗关键词│        │
```

### 4.3 问诊Agent（InterviewAgent）

#### 4.3.1 职责定义

问诊Agent 负责多轮结构化信息采集，将用户口语化描述转化为标准医学术语和结构化病历JSON。

#### 4.3.2 问诊策略设计

```
问诊层次                       采集内容
─────────────────────────────────────────
第一层：主诉精炼     →  核心症状、部位、持续时间
第二层：现病史       →  起病方式、性质、程度、诱因
第三层：伴随症状     →  相关系统症状筛查
第四层：既往史       →  慢性病、手术史、用药史
第五层：过敏史       →  药物/食物过敏
第六层：个人/家族史  →  生活习惯、家族遗传病
```

#### 4.3.3 提问策略约束

| 约束项 | 规则 |
|--------|------|
| 每次提问数 | ≤ 3个问题，避免信息过载 |
| 问题去重 | 基于已收集字段自动过滤重复问题 |
| 动态调整 | 根据用户回答决定下一层提问方向 |
| 终止条件 | 核心字段采集完成 OR 轮次≥5轮 OR 用户主动结束 |
| 非机械追问 | 每次提问前校验上下文，生成自然过渡语 |

### 4.4 问诊Agent Twin-Track 并行架构

#### 4.4.1 职责定义

问诊Agent 是工作流中最复杂的节点，采用 **Twin-Track 并行模型**：Track A（框架驱动信息采集）与 Track B（鉴别搜索）每轮问诊同时触发，搜索不阻塞对话主线程。

#### 4.4.2 Twin-Track 架构

```
┌──────────────────────────────────────────────────────────────┐
│                    问诊Agent (Twin-Track)                      │
│                                                               │
│  输入：用户最新消息 + 已收集 facts                               │
│                                                               │
│  ┌─────────────────────┐    ┌──────────────────────────┐     │
│  │ Track A: 框架驱动     │    │ Track B: 鉴别搜索          │     │
│  │ (主线程, 同步返回)    │    │ (异步, fire-and-forget)     │     │
│  │                      │    │                           │     │
│  │ 1. 提取结构化事实      │    │ 1. 从 collected_info 构造   │     │
│  │ 2. 检查完整性         │    │    多路检索查询              │     │
│  │ 3. LLM 生成追问/确认   │    │ 2. RAG 检索 (向量+关键词)   │     │
│  │ 4. 返回下一轮提问      │    │ 3. 去重 + 累积到共享缓存     │     │
│  └──────────┬──────────┘    └───────────┬──────────────┘     │
│             │                           │                     │
│             └──────────┬────────────────┘                     │
│                        ▼                                      │
│              ┌──────────────────┐                             │
│              │  每轮合并         │                             │
│              │  collected_info   │                             │
│              │ + search_results  │                             │
│              └──────────────────┘                             │
│                        │                                      │
│                        ▼                                      │
│              选项卡片 → 用户交互                               │
└──────────────────────────────────────────────────────────────┘
```

#### 4.4.3 查询改写策略

| 策略 | 示例 | 说明 |
|------|------|------|
| 主诉→检索 | "膝盖酸痛" → "膝盖酸痛 鉴别诊断 临床指南" | 自动附加专业限定词 |
| 症状组合 | 主诉 + 伴随症状 → "膝盖酸痛 肿胀 可能诊断" | 多症状联合检索 |
| 多路并行 | 同时生成2-3个不同角度查询 | 提高召回率 |

#### 4.4.4 异步执行设计

```
每轮 API 调用:
  用户消息 → L0过滤 → interview_node
                         │
            ┌────────────┤
            ▼            ▼
        Track A       Track B (asyncio.create_task)
        (同步)         (不阻塞, 结果写入 _search_results_cache)
            │            │
            ▼            ▼
       下一轮时 _pull_search_results() 拉取累积结果
```

**关键设计：**
- Track B 在 interview_node 每轮处理时以 `asyncio.create_task()` 触发
- Track A 不等待 Track B，立即返回追问
- 后续轮次通过 `_pull_search_results()` 拉取已完成的搜索结果
- 诊断阶段使用累积的 `search_results`

## 5. 协作模式与工作流

### 5.1 三种协作模式

#### 5.1.1 策略模式（Strategy Pattern）

MasterAgent 根据意图动态选择处理策略：

```
                    ┌──────────┐
                    │ 用户输入  │
                    └────┬─────┘
                         │
                   ┌─────▼──────┐
                   │  意图识别   │
                   └─────┬──────┘
                         │
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
   ┌──────────┐   ┌──────────┐   ┌──────────┐
   │ 问诊策略  │   │ 问答策略  │   │ 紧急策略  │
   │          │   │          │   │          │
   │→问诊Agent│   │→搜索Agent│   │→急救指引  │
   │→搜索Agent│   │→直接回答  │   │→终止流程  │
   │→诊断Agent│   │          │   │          │
   └──────────┘   └──────────┘   └──────────┘
```

#### 5.1.2 黑板模式（Blackboard Pattern）

所有Agent通过 LangGraph State 共享上下文，禁止Agent间直接调用：

```
                    ┌──────────────────────┐
                    │   LangGraph State     │
                    │   (共享黑板)           │
                    │                       │
                    │  ┌─────────────────┐  │
                    │  │ messages         │  │  ← 对话历史
                    │  │ collected_info   │  │  ← 结构化病历
                    │  │ search_results   │  │  ← 检索结果
                    │  │ current_stage    │  │  ← 当前阶段
                    │  │ intent           │  │  ← 识别意图
                    │  │ red_flag_raised  │  │  ← 红旗状态
                    │  └─────────────────┘  │
                    └───┬──────┬──────┬─────┘
                  ◄─────┤      │      ├─────►
              读取/写入   │      │   读取/写入
              ┌──────────┐│      │┌──────────┐
              │问诊Agent  ││      ││诊断Agent  │
```

### 5.2 LangGraph 状态图定义

#### 5.2.1 状态节点

```python
from typing import TypedDict, List, Optional
from langgraph.graph import StateGraph, END

class ConversationState(TypedDict):
    """LangGraph 共享状态定义"""
    # 对话基础
    messages: List[dict]              # 完整对话历史
    current_stage: str                # 当前阶段: init/collect/diagnose/done/emergency

    # 意图与路由
    intent: str                       # diagnosis/question/emergency/greeting/status
    route_decision: str               # 路由目标节点

    # 结构化病历 (问诊Agent写入)
    collected_info: dict              # 结构化病历JSON

    # 知识检索 (搜索Agent写入)
    search_results: List[dict]        # 去重知识片段列表
    search_queries: List[str]         # 已执行的检索查询

    # 诊断结果 (诊断Agent写入)
    diagnosis_result: Optional[dict]  # 诊断报告JSON

    # 安全
    red_flag_raised: bool             # 是否触发红旗
    safety_checks_passed: bool        # 安全检查是否通过

    # 元数据
    round_count: int                  # 当前问诊轮次
    max_rounds: int                   # 最大问诊轮次
    session_id: str                   # 会话标识
```

#### 5.2.2 工作流图结构 (v2.0)

```python
# LangGraph 工作流定义
workflow = StateGraph(ConversationState)

# 注册节点 (v2.0: 移除独立 search 节点, 搜索内嵌为 Track B)
workflow.add_node("safety_check", safety_check_node)    # 安全安检入口
workflow.add_node("interview", interview_agent_node)     # Twin-Track 问诊
workflow.add_node("response", response_node)             # 回复生成(含诊断)
workflow.add_node("human_review", human_review_node)     # 人工兜底

# 定义边与条件路由
workflow.set_entry_point("safety_check")

workflow.add_conditional_edges("safety_check", route_by_intent, {
    "interview": "interview",
    "response": "response",
})

workflow.add_conditional_edges("interview", check_complete, {
    "interview": "interview",     # 信息不足, 继续问诊循环
    "response": "response",       # 信息完整/需输出, 进入回复
})

workflow.add_conditional_edges("response", after_response, {
    "done": END,
    "human_review": "human_review",
})

workflow.add_edge("human_review", END)
```

> **v2.0 关键简化**: 移除了独立的 `search`、`diagnosis`、`emergency`、`direct_answer`、`status_report` 节点。这些功能分别内聚到：Track B（搜索）、response_node（诊断/应急/问答）、route_by_intent（路由分流）。

### 8.2 PII 脱敏规则

| 信息类型 | 检测方式 | 脱敏方式 | 示例 |
|---------|---------|---------|------|
| 姓名 | 正则 + NER模型 | 替换为 `[姓名]` | "我叫张三" → "我叫[姓名]" |
| 身份证号 | 正则 (18位/15位) | 替换为 `[身份证号]` | "110101199001011234" → "[身份证号]" |
| 手机号 | 正则 (11位) | 替换为 `[手机号]` | "13800138000" → "[手机号]" |
| 地址 | NER模型 | 替换为 `[地址]` | "北京市朝阳区" → "[地址]" |
| 邮箱 | 正则 | 替换为 `[邮箱]` | "abc@example.com" → "[邮箱]" |

### 8.3 安全事件响应

| 事件 | 响应动作 |
|------|---------|
| 红旗触发 | 立即中断Agent链 → 返回急救指引 → 记录审计日志 → 标记会话为紧急 |
| LLM输出违规 | 拦截输出 → 回退安全回复 → 记录事件 → 触发告警 |
| PII泄露风险 | 自动脱敏 → 日志告警 → 合规审查 |
| 知识引用异常 | 降级至仅返回安全建议 → 引导就医 → 触发知识库审查 |

```sql
-- 会话表
CREATE TABLE sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id VARCHAR(255),
    status VARCHAR(50) DEFAULT 'active',  -- active/paused/completed/emergency_terminated
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    metadata JSONB DEFAULT '{}'
);

-- 对话消息表
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID REFERENCES sessions(id),
    role VARCHAR(20) NOT NULL,           -- user/assistant/system
    content TEXT NOT NULL,
    content_type VARCHAR(50) DEFAULT 'text',
    created_at TIMESTAMP DEFAULT NOW()
);

-- 结构化病历表
CREATE TABLE medical_records (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID REFERENCES sessions(id) UNIQUE,
    record_data JSONB NOT NULL,          -- 结构化病历JSON
    version INTEGER DEFAULT 1,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 审计日志表
CREATE TABLE audit_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id UUID REFERENCES sessions(id),
    agent_name VARCHAR(100),             -- master/interview/search/diagnosis
    event_type VARCHAR(100),             -- intent_detect/question_generate/search_execute/diagnosis_generate
    input_data JSONB,
    output_data JSONB,
    token_usage JSONB,                   -- {prompt_tokens, completion_tokens, total_tokens}
    latency_ms INTEGER,
    model_name VARCHAR(100),
    red_flag_triggered BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 知识库元数据表
CREATE TABLE knowledge_entries (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title VARCHAR(500),
    source VARCHAR(255),
    source_type VARCHAR(50),             -- guideline/consensus/textbook/review/case_report
    publish_year INTEGER,
    content TEXT,
    embedding_id VARCHAR(255),           -- pgvector (PostgreSQL 扩展)中的向量ID
    authority_score FLOAT DEFAULT 0.5,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_sessions_user ON sessions(user_id);
CREATE INDEX idx_sessions_status ON sessions(status);
CREATE INDEX idx_messages_session ON messages(session_id);
CREATE INDEX idx_medical_records_session ON medical_records(session_id);
CREATE INDEX idx_audit_logs_session ON audit_logs(session_id);
CREATE INDEX idx_audit_logs_agent ON audit_logs(agent_name);
CREATE INDEX idx_audit_logs_created ON audit_logs(created_at);
```
